Animated operating flow

How every app manages order, care, rider and exception flow

Use this slide as the main management explanation: one customer action creates a controlled chain across supply, rake manager, rider, customer care and central command.

C Customer App

Passenger places order, checks tracking, invoice, feedback and complaint.

O Order Created

Order enters live database with train, coach, berth, item and delivery slot.

P Train Pantry

If item is available onboard, rake manager accepts and prepares directly.

B Base Kitchen App

Receives order/indent, accepts within time, prepares, marks ready and invoices.

R RK Partner App

Approved partner supplies when route/station/catalog rule permits.

M Rake Manager Command Center

Controls accept, prepare, initiate delivery, dispute, cancel and complaint status.

D Rider App

Pickup, OTP, cash collection if COD, delivered confirmation and proof.

CC Customer Care App

Sees complaint, calls passenger/vendor, updates status and resolution note.

F Feedback / Complaint

Passenger issue routes to rake manager and customer care for action.

T Tracking

Passenger sees accepted, preparing, out-for-delivery, delivered or cancelled.

CM Central Command

Management sees live orders, GMV, margin, rider status, complaints and risk.

N If No / Failed

No stock, partial quantity, rider delay, COD issue or unresolved complaint moves to alternate source, refund/cancel, dispute or escalation.

Revenue creation

Scale target revenue model: 200 RK trains

Business volume target: cross 5,000 daily orders in 3 months and 20,000 daily orders in 6 months.

Milestone Orders/day Orders/train/day AOV Daily GMV Monthly GMV Monthly contribution
Pilot start 1,000 5 Rs 160 Rs 1.6L Rs 4.8Cr Rs 28.8L @ 6%
Month 3 target 5,000 25 Rs 180 Rs 9.0L Rs 27.0Cr Rs 3.24Cr @ 12%
Month 6 target 20,000 100 Rs 190 Rs 38.0L Rs 114.0Cr Rs 17.1Cr @ 15%

Calculation basis: 200 trains, monthly GMV = orders/day x AOV x 30 days. Monthly contribution uses assumed net margin. Additional upside can come from partner commission, promoted menu placement, catering analytics and route services.

Technical architecture

Pilot now, production-ready path next

Version 1 is intentionally lean: static apps plus Firebase live sync. Version 2 hardens identity, security and audit.

User apps

Customer, command center, base kitchen, partner, delivery, control tower and central command.

Web hosting

GitHub Pages static deployment for controlled pilot.

Data layer

Firebase Firestore live collections: orders, feedbacks, indents, invoices and masters.

Security path

Firebase Auth, role claims, App Check and strict Firestore rules in production.

Integration path

Secure backend/API proxy for PNR, WhatsApp/SMS, payment and reporting database.

PNR lookup should be protected through backend/API proxy14
